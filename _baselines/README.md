# Canonical calibration baselines

This directory ships **pre-computed baselines** for the patterns whose calibration data is deterministic — i.e. patterns where the metrics don't depend on an LLM call. Today that's the Span-of-Control pattern (#34); its math is gated to Python, so the baseline JSON is reproducible.

For the LLM-bearing 33 patterns, the calibration data depends on the LLM's actual scoring of the canonical trace. We can't ship those without burning an API budget at build time and locking the result to one model + one moment in time. Instead, this README documents the **recipe** for users to generate their own pattern baselines once, then use `vstack-config set baselines-dir` + the pattern's `baseline_path` arg to compare future runs against them.

## What's pre-shipped

| File | Pattern | Crew shape | What it represents |
|---|---|---|---|
| `canonical/span_of_control_small_flat.json` | #34 Span-of-Control | 1 orchestrator + 2 workers, 10 req/min | **Healthy small-crew baseline.** All metrics in the "moderate" band; no bottleneck. |
| `canonical/span_of_control_two_layer.json` | #34 Span-of-Control | 1 orchestrator → 3 leads → 9 workers, 50 req/min | **Healthy mid-scale baseline.** Two-layer hierarchy. |
| `canonical/span_of_control_hub_and_spoke.json` | #34 Span-of-Control | 1 orchestrator + 12 flat workers, 100 req/min | **Failure-mode baseline.** Centralized orchestrator under load — the textbook crew shape this pattern was built to detect. Useful as a "negative example" reference point. |

These are the three crew topologies most teams will ship one of. If your crew matches one of them, point the pattern at the matching baseline:

```python
from vstack.span_of_control import SpanLoadCalculator
detection = SpanLoadCalculator(llm).run(
    trace,
    baseline_path="_baselines/canonical/span_of_control_two_layer.json",
)
```

The `BaselineComparison` field on the returned detection will then carry deltas: which metric drifted, by how much, with what significance.

## Recipe: building your own per-pattern baseline

For any of the LLM-bearing patterns, the workflow is:

1. **Pick a canonical "healthy" trace.** A recent run on a crew that's working the way you want. Capture the input model (agent trace, multi-agent trace, conversation trace, etc.) in JSON.
2. **Run the pattern in `forensic` mode.** Higher fidelity = better reference point.
3. **Save the detection JSON** into `~/.vstack/baselines/<pattern_name>.json` (the location `vstack-config path baselines` prints).
4. **Future runs auto-detect the baseline.** Pass `baseline_path` explicitly or let the pattern's CLI use the default path.

Example:

```bash
# Step 2-3 in one shot via the pattern's CLI:
vstack-lewin analyze --trace healthy_run.json --mode forensic \
    --baseline-out ~/.vstack/baselines/lewin.json

# Future runs compare automatically:
vstack-lewin analyze --trace new_run.json --mode standard
```

The returned `BaselineComparison.deltas` tells you which dimensions drifted. The pattern's playbook decides which deltas matter — for Lewin, a swing in `dominant_locus` is more important than a 0.1-point score shift.

## Re-generating the canonical baselines

These were generated by `_baselines/scripts/generate_canonical.py`. Re-run when:

- A Span-of-Control metric formula changes (the JSONs encode the *current* output of the calculator; a formula bump invalidates them).
- A new canonical crew shape is added.
- The Pydantic model schema gains a required field.

```bash
python _baselines/scripts/generate_canonical.py
```

The script is committed alongside the baselines so the regeneration step is reproducible and the canonical traces are visible in source control. CI does **not** run the script automatically — that's intentional: drift in the JSONs (from a Pydantic model bump or a metrics tweak) should be a deliberate commit, not a silent migration.
